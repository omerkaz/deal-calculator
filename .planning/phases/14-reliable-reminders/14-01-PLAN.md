<objective>
Add an `email_send_log` tracking table to the database schema so cron reminder
functions can record when each email was sent and skip patients who already
received their reminder — enabling at-least-once delivery (MAIL-02).

Purpose: Replace the fragile 24h BETWEEN windows (D017) with a durable
"sent or not?" check that survives missed cron runs.

Output: Updated `supabase/schema.sql` with the new table, RLS, and index.

Design Decision — Tracking Table over Columns:
A dedicated `email_send_log` table is chosen over adding per-feature columns to
`patients` because:
1. There are 6 current reminder features + 4 incoming DRIP features (Phase 15)
2. A table scales to new features without schema changes
3. It provides an audit trail of when emails were actually sent
4. The NOT EXISTS subquery pattern is clean and efficient
5. Re-engagement (cold → lead) is handled naturally: the query checks
   `sent_at > COALESCE(patients.state_changed_at, patients.created_at)`
   so old log entries from a prior lifecycle run don't suppress new sends
</objective>

  <action>
Add a new section after the practitioner_settings RLS policies (before the
state_changed_at migration block) with the heading:

  -- ── Email send tracking log ──

Create table `email_send_log`:
- `id` uuid PK default gen_random_uuid()
- `patient_id` uuid NOT NULL, FK → patients(id) ON DELETE CASCADE
- `feature` text NOT NULL — stores the feature key (e.g. 'blood_test_reminder',
  'lead_day3', 'week_6_checkin', 'end_review', 'lead_day7', 'lead_day12')
- `sent_at` timestamptz NOT NULL DEFAULT now()

Create index `idx_email_send_log_patient_feature` on (patient_id, feature) for
efficient NOT EXISTS lookups from cron functions.

Enable RLS on the table. Add a single SELECT policy:
  "Users can view email send log" — for SELECT using
  (patient_id IN (SELECT id FROM patients WHERE created_by = auth.uid()))
  This lets the practitioner see the log via the app if needed in the future.

No INSERT/UPDATE/DELETE policies for the practitioner — only SECURITY DEFINER
cron functions write to this table (they bypass RLS).

Do NOT add a UNIQUE constraint on (patient_id, feature) — a patient may
legitimately receive the same feature's email in different lifecycle runs
(e.g. cold → lead re-engagement). Deduplication is handled by the NOT EXISTS
query pattern in the cron functions (plan 14-02).
